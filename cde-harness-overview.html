하네스란?

CDE AI Harness는 AWS Solutions Architect가 고객 데모용 프로토타입을 빠르게 만들 수 있도록 설계된 Claude Code 기반 자동화 프레임워크다. 회의록·요구사항 문서·다이어그램 같은 비정형 입력을 받아, 도메인 리서치 → 요구사항 분석 → 아키텍처 설계 → 스펙 작성 → 코드 생성 → QA → 보안 감사 → 인수인계까지 단계별 서브에이전트가 순차적으로 실행된다.

슬래시 커맨드 (8개)

사용자가 Claude Code에서 직접 실행하는 진입점. 각 커맨드는 Phase 게이트와 APPROVAL GATE를 따라 단계별로 진행한다.

/brief PRE-PIPELINE

.pipeline/input/raw/ 안의 회의록·다이어그램·문서를 분석해 표준 customer-brief.md 자동 생성

/pipeline FULL RUN

customer-brief.md 부터 핸드오버 직전까지 전체 파이프라인 실행 (Stage 1 → 6b)

/iterate FEEDBACK LOOP

고객 피드백 영향 범위 분석 + 최소 재생성. iterate/v{N+1} 브랜치 위에서 진행

/reconcile REVERSE SYNC

바이브코딩·버그픽스 후 src/ 코드 → 아티팩트 역동기화. --qa 옵션으로 품질 루프 재실행

/awsarch MOCK → AWS

InMemoryStore 프로토타입을 실제 AWS(DynamoDB · S3 · Cognito)로 전환. CDK 인프라 + 데이터 레이어 교체

/handover FINAL

고객 개발팀 인수인계용 패키지 (아키텍처 문서 · API · 환경 설정 · 프로덕션 전환 체크리스트)

/pipeline-from RESUME

특정 스테이지부터 파이프라인 재개. 이전 아티팩트는 그대로 활용

/pipeline-status INSPECT

현재 파이프라인 진행 상태 · 버전 · 아티팩트 요약 표시

/iterate vs /reconcile — 두 가지 변경 흐름

프로토타입은 만들어진 후에도 계속 변한다. 변화의 출발점이 고객 피드백(요구사항)이냐, 아니면 SA의 직접 코드 변경이냐에 따라 두 갈래의 흐름이 존재한다.

방향성 비교

/iterate TOP-DOWN
고객 피드백(입력) → 코드
회의록·이메일·새 요구사항이 raw/에 추가되면, 영향 범위를 분석한 뒤 requirements-analyst부터 파이프라인을 다시 흐른다.
/reconcile BOTTOM-UP
코드 변경 → 아티팩트
SA가 src/를 직접 손본 후(바이브코딩·버그픽스), 변경된 코드를 기준으로 스펙·아키텍처·요구사항을 역동기화한다.
/iterate 고객 피드백 기반 반복 개선

새로운 피드백·요구사항을 .pipeline/input/raw/에 추가한 뒤 실행한다. 영향 범위 분석 → 모호한 부분 질의 → brief 갱신 → 파이프라인 재실행까지 한 번에 처리한다.

Phase 흐름

Phase 0
브랜치 선행 생성 — 분석 산출물이 main에 남지 않도록 iterate/v{N+1} 브랜치를 먼저 만든다. (git-manager · pre-iterate)
Phase 1
변경 감지 + 영향 분석 + clarifications 생성 — feedback-analyzer가 raw/ 새 파일과 기존 아티팩트(요구사항·아키텍처·스펙·코드)를 비교하여 영향 범위를 산출. 모호한 피드백은 clarifications.md에 질문 형태로 자동 작성. APPROVAL GATE.
Phase 2
clarifications 답변 대기 (조건부) — clarifications.md가 생성된 경우만 실행. 사용자가 답변을 채우거나 "추론값으로 진행"을 선택할 때까지 대기. 답변이 없으면 brief-composer가 ## Assumptions로 추론 근거를 기록.
Phase 3
입력 파일 갱신 — brief-composer가 customer-brief.md / source-analysis.md / manifest.json 3개 파일을 갱신. 기존 요구사항은 보존하고 변경/추가분만 통합.
Phase 4
state.json 버전 추가 + 아티팩트 디렉토리 생성 — v{N+1} 엔트리 추가. v{N}의 00-domain/은 도메인 지식이 변하지 않으므로 그대로 복사.
Phase 5
파이프라인 재실행 (requirements-analyst부터) — domain-researcher는 건너뛰고 Stage 2~7을 순차 실행. 영향이 작아도 항상 requirements부터 시작해 FR → 아키텍처 → 스펙 → 코드 추적 체인을 유지.
⚠️ 핵심 설계 의도
feedback-analyzer가 recommended_reentry를 산출해도 항상 requirements부터 재실행. 중간 문서를 건너뛰면 추적성이 끊어진다.
🔄 Quality Loop
Phase 5의 QA(3회) → Review(2회) → Security(1회) 루프 한도 초과 시 halted 상태로 진입. halt-report.md와 함께 3가지 복구 옵션 제시.

전형적인 사용 시나리오

  • 고객 데모 후 받은 피드백 메일을 raw/feedback-2026-05-17.md로 저장 → /iterate
  • 새 페이지·기능 추가 요청을 회의록으로 정리 → raw/에 투입 → /iterate
  • 기존 요구사항의 우선순위 변경, NFR 강화(접근성·성능 등) 반영
/reconcile 코드 변경 후 아티팩트 역동기화

SA가 파이프라인 밖에서 src/를 직접 수정한 뒤(바이브코딩·긴급 버그픽스·실험적 변경) 실행한다. 변경된 코드를 분석하여 요구사항·아키텍처·스펙·생성 로그를 코드 현재 상태에 맞춰 갱신한다.

두 가지 모드

/reconcile
docs-only · 경량
Phase 1~5만 실행. 아티팩트만 동기화. src/ 코드는 절대 수정 안 함.
/reconcile --qa
docs-qa · 전체
Phase 1~6 실행. 동기화 후 QA/Review/Security 재실행. 품질 루프 실패 시에만 코드 수정 허용.

Phase 흐름

Phase 1
변경 감지 + 역방향 영향 분석 (읽기 전용) — reconcile-analyzer(analyze)가 git 이력과 src/ 현재 상태를 비교하여 변경을 분류(structural 또는 refinement). 영향받는 아티팩트 목록 산출. APPROVAL GATE.
Phase 2
브랜치 생성 — git-manager(pre-reconcile)가 reconcile/v{N+1} 브랜치 생성. Phase 1의 revisions 산출물을 먼저 커밋한 후 분리.
Phase 3
아티팩트 동기화 — 4개 sub-phase로 나누어 코드 → 아티팩트 역방향 갱신.
3a. 생성 로그 (04-codegen/) — ad-hoc 파일에 "origin": "ad-hoc", 수정된 파일에 "status": "reconciled" 마커
3b. 스펙 (03-specs/) — backend-spec.json, frontend-spec.json, _manifest.json의 requirements_coverage 갱신
3c. 아키텍처 (02-architecture/) — src/ 디렉토리 스캔으로 architecture.json 갱신, FR 역매핑
3d. 요구사항 (01-requirements/) — structural 변경일 때만. 새 FR에 "origin": "reconcile" 마커
Phase 4
state.json 갱신 + reconcile-report.md 생성 — baseline_commit/head_commit, change_classification, 갱신 아티팩트 목록을 한국어 리포트로 작성.
Phase 5
완료 (docs-only) 또는 QA 진행 (--qa) — git-manager(post-reconcile)로 아티팩트와 리포트만 커밋. src/ 코드는 커밋하지 않는다 (이미 사용자가 ad-hoc으로 커밋했기 때문).
Phase 6
QA + Review + Security (--qa 전용) — 갱신된 requirements/architecture를 기준으로 현재 코드를 검증. 이때만 code-generator-* 에이전트의 코드 수정이 허용된다.

변경 분류 로직

structural
새 페이지/API/도메인 모델 추가, 기존 컴포넌트 분리·통합, 데이터 흐름 변경 → 요구사항(01-requirements/)까지 전부 갱신
refinement
버그 픽스, 스타일 조정, 성능 개선, 리팩토링 → 요구사항은 v{N}에서 그대로 복사하고 스펙·아키텍처만 갱신

전형적인 사용 시나리오

  • 데모 직전 긴급 버그픽스로 src/components/VehicleTable.tsx를 직접 수정 → /reconcile
  • 실험적으로 새 페이지를 바이브코딩으로 추가 → 검증 필요 시 /reconcile --qa
  • 코드 리팩토링 후 스펙 문서가 어긋나는 경우 동기화

언제 무엇을 쓸까?

데모 직전 코드를 직접 패치했다 — ✓

실패 시 main 브랜치 보호 iterate/v{N+1} reconcile/v{N+1}

/awsarch & /handover — 실서비스 전환 & 인수인계

프로토타입이 충분히 무르익으면 실제 AWS 리소스로 전환하고(/awsarch), 고객 개발팀에 인수인계할 문서 패키지를 만든다(/handover).

/awsarch Mock 프로토타입 → 실제 AWS 리소스 전환

InMemoryStore 기반 프로토타입을 실제 AWS(DynamoDB · Aurora · S3 · Cognito · Bedrock AgentCore)로 전환한다. CDK TypeScript 인프라를 생성하고, 듀얼 모드 데이터 레이어를 구현하여 DATA_SOURCE 환경 변수만으로 mock/real을 전환할 수 있게 한다.

세 가지 모드

/awsarch --plan
설계만 · 배포 없음
Phase 1까지만. aws-architecture.json + 비용 추정 산출. 의사결정 자료로 활용.
/awsarch
전체 · 기본
Phase 1~4. 설계 + CDK 배포 + 데이터 레이어 교체 + 시드 마이그레이션.
/awsarch --qa
전체 + 품질 검증
Phase 1~5. AWS 통합 후 QA/리뷰/보안 재실행. AWS 특화 보안 점검 포함.

Phase 흐름

Pre-flight
사전 점검 + 브랜치 생성 — completed 버전 존재 / src/lib/db/store.ts 존재 / npm run build 성공 / aws sts get-caller-identity로 자격 증명 확인. awsarch/v{N+1} 브랜치 생성.
Phase 1
AWS 인프라 설계 — aws-architect가 architecture.json + requirements.json + src/types/ + src/data/seed.ts를 분석하여 aws-architecture.json 산출. services / iam_policies / cost_estimate 포함. APPROVAL GATE — 비용 확인 (모든 모드에서 실행).
Phase 2
CDK 코드 생성 + 데이터 레이어 교체 — aws-deployer가 infra/ 디렉토리 + DataStore 인터페이스 + DynamoDBStore 구현 + store-factory를 생성. 기존 repository와 API route는 await 추가로 수정.
검증: npm run build + cd infra && npx tsc --noEmit 성공
Phase 3
CDK 배포 — bootstrap → diff → APPROVAL GATE (cdk diff 확인) → deploy. --require-approval broadening으로 IAM 권한 확대 시 추가 확인. cdk-outputs.json을 .env.local로 변환.
Phase 4
시드 데이터 마이그레이션 + 듀얼 모드 검증 — src/data/seed.ts 데이터를 DynamoDB로 삽입. 두 모드 모두 빌드 통과 검증:
npm run build (DynamoDB 모드, 기본) + DATA_SOURCE=memory npm run build (mock 모드 여전히 동작)
Phase 5
QA + Review + Security 재실행 (--qa 전용) — DATA_SOURCE=dynamodb 환경에서 E2E 재실행. reviewer는 10번째 카테고리 (AWS 통합 품질) 추가 점검. security-auditor는 IAM 과도 권한 / 하드코딩 자격 증명 / .env.local의 .gitignore 포함 여부 등 AWS 특화 항목 검사.

class InMemoryStore implements DataStore { /* mock */ }

function createStore(): DataStore {

    ? new DynamoDBStore()
    : new InMemoryStore(); // 기본값

repository 인터페이스는 동일하므로 고객 데모 시 오프라인 환경에서도 mock 모드로 시연 가능. AWS 비용을 발생시키지 않고 개발할 수 있다.

핵심 원칙

PRINCIPLE 01

Phase Gate · APPROVAL GATE

사용자 승인 없이 다음 Phase로 진행 금지. CHECKPOINT 통과 시에만 진행.

PRINCIPLE 02

에이전트만 코드 수정

/pipeline · /iterate · /awsarch 시 src/, infra/는 code-generator·aws-deployer만 수정한다.

PRINCIPLE 03

AI 기능은 Mocking 금지

반드시 Amazon Bedrock + Strands SDK로 실제 동작. @aws-sdk/client-bedrock-runtime 직접 호출 금지.

PRINCIPLE 04

버전 격리

.pipeline/artifacts/v{N}/ 디렉토리로 모든 산출물을 버전별 격리. 이전 버전은 보존된다.

PRINCIPLE 05

Quality Loop

qa-engineer(테스트) → reviewer(품질) → security-auditor(OWASP) PASS까지 자동 반복. budget으로 무한루프 차단.

PRINCIPLE 06

언어 정책

.md = 한국어 · .json = 영어 · 코드 식별자 = 영어 · 주석 설명 = 한국어 · 사용자 보고 = 한국어.

PRINCIPLE 07

SSOT · state.json 보호

checkpoint.mjs 서브커맨드가 state.json 유일 writer. PreToolUse hook이 Write/Edit/redirect/sed/python/node 우회 12종 차단.

PRINCIPLE 08

--auto 안전 게이트 5종

/awsarch 비용 · cdk destroy · /iterate · Circuit Breaker · 보안 critical은 --auto로 우회 불가. SSOT: .claude/policies/auto-safety-gates.md.

PRINCIPLE 09

AI 모델 ID 화이트리스트 (Rule 13)

haiku-4-5 · sonnet-4-6 · opus-4-7 3개만 허용. 도구/에이전트 단위로 코드에 직접 박는다. process.env.BEDROCK_MODEL_ID fallback 금지.
